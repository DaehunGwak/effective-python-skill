# Effective Python — 코드 품질 스킬

"Effective Python: 더 나은 Python을 위한 125가지 구체적인 방법"
(Brett Slatkin, 3판)의 원칙을 적용하여 관용적이고, 유지보수하기 쉽고,
고성능인 Python을 생성합니다.

> **저작권 안내**: 이 스킬은 책에서 *영감을 받은 원칙과 패턴*을 독자적인 표현으로
> 인코딩합니다. 책의 원문을 재현하지 않습니다. 전체 설명과 맥락을 위해
> 책 구매를 권장합니다.

## 이 스킬의 작동 방식

이 스킬은 챕터별 레퍼런스 파일에 대해 **점진적 공개**를 사용합니다.
지금 읽고 있는 SKILL.md에는 다음이 포함되어 있습니다:
1. 코드 관심사 → 레퍼런스 파일을 매핑하는 라우팅 테이블
2. 모든 Python 코드에 적용되는 범용 원칙
3. 종합적인 코드 감사를 위한 리뷰 체크리스트

**레퍼런스 파일**은 `references/ko/`에 챕터별 상세 가이드를 포함합니다.
현재 작업에 관련된 레퍼런스 파일만 로드하세요.

## 범용 원칙 (항상 적용)

이 원칙들은 모든 챕터를 관통하며, 작성하는 모든 Python 코드에 반영되어야 합니다:

### 1. 명확함이 영리함보다 우선
의도를 전달하는 코드를 작성하세요. 한 줄짜리 코드가 이해하기 위해 머리를 굴려야 한다면,
이름이 있는 중간 단계로 분리하세요. 서술적인 이름을 가진 도우미 함수가
복잡한 표현식보다 거의 항상 낫습니다.

### 2. 최신 Python (3.10+) 관용 표현
- `Union[X, Y]` 대신 `X | Y` 유니온 문법 사용
- `Optional[x]` 대신 `x | None` 사용
- 의도를 명확히 할 때 `match`/`case`로 구조적 패턴 매칭 사용
- `typing` 임포트 대신 네이티브 타입 힌트(`list[str]`, `dict[str, int]`) 사용
- 왈러스 연산자(`:=`)는 중복 계산을 제거할 때 사용 — 과시용이 아님
- 구조화된 데이터에는 원시 튜플/딕셔너리 대신 `@dataclass` 또는 `NamedTuple` 사용

### 3. 빠르게 실패하고, 명확하게 실패하기
- 에러 조건에서 `None`을 반환하는 대신 예외를 발생시키기
- 구체적인 예외 타입을 사용하고, 절대로 bare `except:` 사용하지 않기
- 경계에서 입력을 검증하고, 내부에서는 데이터를 신뢰하기
- 개발 중 불변 조건에는 `assert` 사용

### 4. 상속보다 컴포지션
- 깊은 상속 계층 대신 컴포지션과 위임을 선호
- 수평적 기능 공유에는 믹스인 사용
- 메타클래스 대신 등록 패턴에 `__init_subclass__` 사용
- 인터페이스 정의에 `Protocol` (구조적 타이핑) 사용

### 5. 명시적 리소스 관리
- 리소스(파일, 잠금, 커넥션)에는 항상 `with` 문 사용
- 커스텀 리소스 타입에는 `__enter__`/`__exit__` 구현
- 간단한 경우에는 `contextlib.contextmanager` 사용

## 코드 리뷰 체크리스트

Python 코드 리뷰를 명시적으로 요청받았을 때, 다음 카테고리를 체계적으로 확인합니다.
적용되는 각 카테고리에 대해 관련 레퍼런스 파일을 로드하세요:

1. **파이썬다운 스타일** — 최신 관용 표현이 사용되었는가? (ch01, ch02)
2. **자료구조** — dict/list/set/tuple/dataclass의 올바른 선택인가? (ch04, ch07)
3. **반복** — 가능한 곳에 제너레이터가 사용되었는가? 반복 중 변경은 없는가? (ch03, ch05)
4. **함수 설계** — 명확한 시그니처인가? 키워드 전용 인자? 가변 기본값 없는가? (ch06)
5. **클래스 설계** — 상속의 적절한 사용인가? 컴포지션이 선호되었는가? (ch07, ch08)
6. **동시성** — 스레드 안전한가? async가 유용한 곳에 사용되었는가? (ch09)
7. **견고성** — 적절한 에러 처리인가? 경계에서 방어적 코딩인가? (ch10)
8. **성능** — 명백한 비효율이 있는가? 성급한 최적화는 피했는가? (ch11)
9. **테스트 가능성** — 의존성 주입 가능한가? 가능한 곳에 순수 함수인가? (ch12)
10. **협업** — 독스트링? 타입 힌트? 깔끔한 임포트? (ch13)

발견된 각 이슈에 대해 보고:
- **무엇**: 구체적인 코드 패턴
- **왜**: 왜 중요한지 (정확성, 가독성, 성능, 또는 유지보수성)
- **어떻게**: 구체적인 코드 예제와 함께 권장하는 수정 방법

## 사용 모드

### 자동 모드 (코드 생성 시)
Python 코드를 생성할 때, 위의 범용 원칙을 자동으로 적용하고
작성 중인 코드에 따라 1~2개의 관련 레퍼런스 파일을 로드합니다.
이 스킬을 사용하고 있다고 알리지 마세요 — 그냥 좋은 Python을 작성하세요.

### 리뷰 모드 (`/effective-python [경로]`)
지정된 코드에 대해 전체 체크리스트를 기준으로 종합적인 리뷰를 수행합니다.
관련된 모든 레퍼런스 파일을 로드합니다. 심각도별로 그룹화된 구조화된
리뷰를 출력합니다: 🔴 심각 → 🟡 제안 → 🟢 발견된 우수 사례.

### 리팩토링 모드 (`/effective-python refactor [경로]`)
원칙을 적용하여 기존 코드를 리팩토링합니다. 관련 원칙을 참조하며
변경 전/후를 보여줍니다.
